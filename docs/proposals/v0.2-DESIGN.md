# Akashik v0.2 Design — Level 0 Conformance + Selected Level 1 Operations

Status: **Frozen at the design level.** This document defines what v0.2 is. Implementation may begin after Akashik v0.1's public launch on Sunday 10 May 2026.

v0.2 brings `@akashikprotocol/core` to **full Level 0 conformance** and additionally implements four Level 1 operations that complete natural lifecycle stories: `DEREGISTER`, `RETRACT`, `SUPERSEDE`, and `DETECT in list mode` (surfaced as the new `reckon()` method).

This document is the authoritative reference. Every type, contract, and decision below is final at the design level.

---

## Conformance declaration

**v0.2 conforms to Level 0 of the Akashik Protocol Specification.** Additionally, v0.2 implements the following Level 1 operations:

- DEREGISTER (Level 1 #5) — pairs naturally with REGISTER
- DETECT in list mode (Level 1 #6) — surfaced as `reckon()`
- RETRACT — completes Level 0's retracted/superseded ATTUNE filter (Level 0 #9)
- SUPERSEDE — same reasoning as RETRACT

v0.2 does NOT implement other Level 1 requirements: persistent storage, append-only event log, full Lamport logical clocks (v0.2 uses a simpler monotonic counter), confidence scoring on RECORD, since_epoch polling, full capability negotiation.

A full implementation reaches Level 1 in v0.3.

---

## Design principles

**1. v0.1 contracts are immutable at the type level.** Every existing v0.1 method signature, type, and error code is preserved. v0.2 is purely additive at the API surface. The single behaviour change is `attune()`'s ordering — v0.2 returns relevance-sorted results instead of write-order — but the return type is unchanged.

**2. The protocol decides relevance and detects disagreement; the agent decides resolution.** ATTUNE surfaces what's relevant. RECKON surfaces conflicts. Neither resolves anything.

**3. Mechanical now, semantic later.** v0.2's relevance scoring is weighted feature-based (no LLM, no embeddings). v0.2's conflict detection is mechanical key-value comparison. Semantic versions of both arrive in v0.4 (Level 2).

## What ships in v0.2

### Standard message envelope (Level 0 #11)

Wire-format-only. The SDK's TypeScript API stays unchanged from v0.1; envelope construction happens internally.

```ts
type Message = {
  id: string;                          // ULID, message-unique
  type: "REGISTER" | "DEREGISTER" | "RECORD" | "DRAFT" | "COMMIT"
      | "DISCARD" | "RETRACT" | "SUPERSEDE" | "ATTUNE" | "RECKON";
  sender: string;                      // agent id
  epoch: number;                       // field's logical clock at send time
  timestamp: number;                   // Unix ms wall clock
  payload: Record<string, unknown>;    // operation-specific
  correlation_id?: string;             // for request/response pairing
};
```

Encoding is JSON. Protocol version is negotiated at REGISTER time, not stamped on every message.

### REGISTER and DEREGISTER

```ts
field.register({
  id: "researcher-prod-1",
  role: "researcher",
  capabilities?: string[],
}): Promise<RegisterResult>;

field.deregister({
  id: "researcher-prod-1",
}): Promise<void>;
```

REGISTER is optional but recommended. v0.1 code that doesn't call REGISTER continues to work. Idempotent for the same id. Returns `field_capabilities`, `field_protocol_version`, optional `session_id`.

DEREGISTER ends the session; drafts owned by the deregistering agent persist beyond their author's session.

### Auto-generated entry fields

```ts
type FieldEntry = {
  id: string;                          // ULID, unchanged from v0.1
  timestamp: number;                   // Unix ms wall clock, unchanged from v0.1
  epoch: number;                       // NEW: field-wide monotonic counter
  agent?: string;                      // unchanged from v0.1
                                       //   (= "source" in protocol wire vocabulary)
  status: "committed" | "draft" | "retracted" | "superseded";   // NEW
  entry: Record<string, unknown>;      // unchanged from v0.1
  intent: string;                      // unchanged from v0.1
};
```

`epoch` is a monotonic counter starting at 0, incremented on every state-mutating operation. `agent` (SDK API) and `source` (wire format) refer to identical data.

### Updated ATTUNE

```ts
type AttuneContext = {
  agent: string;                       // unchanged from v0.1
  role?: string;                       // NEW: optional in v0.2 for backward compat
  topic?: string;                      // unchanged from v0.1
  max_units?: number;                  // NEW: defaults to 100
};

type FieldEntryWithRelevance = FieldEntry & {
  relevance_score: number;             // 0.0 to 1.0
  relevance_reason: RelevanceReason;
};

type RelevanceReason = {
  components: {
    topic: number;
    role: number;
    recency: number;
    intent: number;
  };
  summary: string;
};

attune(context: AttuneContext): Promise<FieldEntryWithRelevance[]>;
```

**ATTUNE contract changes from v0.1:**

- Return type widens additively. v0.1 consumers reading `id`, `timestamp`, `entry`, `intent`, `agent` find them unchanged.
- **Ordering changes from write-order to relevance-score descending, with epoch descending as tiebreaker.** This is a behaviour change.
- `max_units` defaults to 100; truncates lowest-relevance entries first.
- Drafts excluded unless calling agent is the draft's author.
- Retracted and superseded entries excluded.

**Level 0 reference relevance algorithm:**

```
score = 0.0

if context.topic === entry.topic:
    score += 0.6

if context.role === writer's role:
    score += 0.2

recency_factor = 1.0 - (epoch_distance_from_newest / total_relevant_entries)
score += 0.15 * recency_factor

if entry.intent.length >= 50:
    score += 0.05

score = min(score, 1.0)
```

Implementations conforming to Level 0 may use this algorithm or a similarly weighted alternative.

### `reckon()` — DETECT in list mode

```ts
type ReckonResult = {
  entries: FieldEntryWithRelevance[];
  conflicts: Conflict[];
};

type Conflict = {
  a: FieldEntry;
  b: FieldEntry;
  keys: string[];
};

reckon(context: AttuneContext): Promise<ReckonResult>;
```

A conflict exists between entries A and B when:

1. `entry.topic` matches by strict `===`
2. They share at least one *other* key in `entry` (excluding `topic`)
3. Both values at that shared key are primitives (string, number, boolean, null) and differ

Object-valued shared keys do NOT produce conflicts in v0.2. Conflicts surface only between entries visible to the caller. Pairs appear once, never twice.

### Draft mode operations

```ts
field.draft({
  entry: Record<string, unknown>,
  intent: string,
  agent?: string,
}): Promise<{ draft_id: string }>;

field.commit({
  draft_id: string,
}): Promise<{ id: string; epoch: number; timestamp: number }>;

field.discard({
  draft_id: string,
  intent: string,
  agent?: string,
}): Promise<void>;

read(query?: ReadQuery, options?: { caller?: string }): Promise<FieldEntry[]>;
```

**Scratchpad model:** drafts are private to the writing agent. They do not appear in attune or reckon for other agents. `read({}, { caller })` returns drafts only when caller matches the draft's author.

`commit()` updates `status` from `draft` to `committed`, updates `timestamp` and `epoch` to commit time, returns the same id (commit doesn't change identity).

`discard()` sets `status` to `retracted`. Functionally identical to retracting a never-public entry.

Drafts cannot be edited in place. To revise: discard and create a new one.

### `retract()` and `supersede()`

```ts
field.retract({
  id: string,
  intent: string,                       // 10-char minimum, same as write intent
  agent: string,
}): Promise<void>;

field.supersede({
  superseding_id: string,
  entry: Record<string, unknown>,
  intent: string,
  agent: string,
}): Promise<{ id: string; epoch: number; timestamp: number }>;
```

**RETRACT contract:** only the original writer can retract. Mismatch produces `RETRACT_NOT_AUTHORIZED`. Idempotent on already-retracted entries.

**SUPERSEDE contract:** any registered agent can supersede. Sets predecessor's `status` to `superseded`. Chains extend; only the latest in any chain has `committed` status. ATTUNE and reckon see only the latest.

### Error codes

```ts
type AkashikErrorCode =
  | "INTENT_REQUIRED"          // v0.1
  | "INTENT_TOO_SHORT"         // v0.1
  | "INVALID_ENTRY"            // v0.1
  | "INVALID_QUERY"            // v0.1
  | "AGENT_REQUIRED"           // v0.1
  | "RETRACT_NOT_AUTHORIZED"   // NEW v0.2
  | "ENTRY_NOT_FOUND"          // NEW v0.2
  | "DRAFT_NOT_FOUND";         // NEW v0.2
```

## What v0.2 explicitly does NOT do

- No persistent storage. In-memory only. Persistence is Level 1, deferred to v0.3.
- No append-only event log. Level 1, deferred.
- No full Lamport clocks. v0.2 uses a simple monotonic counter.
- No confidence scoring on RECORD. Level 1, deferred.
- No `since_epoch` polling. Level 1, deferred.
- No vector embeddings or semantic relevance. Level 2, deferred to v0.4+.
- No SUBSCRIBE, HANDOFF, REPLAY, COMPACT. Level 2 or Level 3.
- No authentication or authority hierarchy. Level 3, deferred.
- No coordination extension (COORDINATE, SESSION). Level 3, deferred.
- No deep object matching in conflicts. Deferred to v0.4.
- No in-place draft editing. Discard + redraft is the v0.2 pattern.
- No transport bindings beyond in-process.

## Backward compatibility

| v0.1 capability | v0.2 status |
| --- | --- |
| `createField()` signature | unchanged |
| `field.write()` signature | unchanged; produces `committed` entries |
| `field.read()` signature | widens with optional `options.caller` |
| `field.attune()` signature | widens with optional `role`, `max_units`; return type widens additively to `FieldEntryWithRelevance[]`; **ordering changes from write-order to relevance-sorted** |
| `FieldEntry` shape | three new fields (`epoch`, `status`); existing fields unchanged |
| `AkashikError` class | unchanged |
| `AkashikErrorCode` union | three new codes added; existing codes unchanged |

**One genuine behaviour change:** ATTUNE's result ordering. v0.1 returned write-order. v0.2 returns relevance-sorted. The contract didn't promise write-order specifically, so this is technically not a breaking contract change, but it IS a behaviour change v0.1 consumers might notice. The launch essay acknowledges this as the one place v0.2 evolves v0.1's behaviour.

## Test contract summary

The complete test surface for v0.2. Each item is a contract.

**Backward compatibility:** every v0.1 test in `tests/write.test.ts` and `tests/read.test.ts` passes unchanged. v0.1's `tests/attune.test.ts` is updated for relevance-sorted ordering. `tests/exports.test.ts` updated for new exports.

**Message envelope:** every operation produces a valid envelope. Versioning negotiated at REGISTER.

**REGISTER / DEREGISTER:** returns capabilities and version. Idempotent per id. Empty/whitespace ids and roles rejected.

**Auto-generated fields:** every entry has id, timestamp, epoch, status. `epoch` increments monotonically. `status` defaults appropriately per operation.

**ATTUNE:** sorted by `relevance_score` descending, then `epoch` descending. `max_units` truncates lowest-scoring first. Default 100. Drafts visible only to author. Retracted and superseded excluded. Self-writes excluded. `relevance_score` in [0, 1]. `relevance_reason.components` sum approximately equals `relevance_score`.

**RECKON:** same visibility rules as ATTUNE. Conflicts only between entries in `result.entries`. Conflict on shared primitive-valued keys, same topic. Object-valued keys don't conflict. Pairs appear once. Self-conflicts impossible.

**Draft lifecycle:** `draft()` produces `status: "draft"`. Drafts visible to author only. `commit()` transitions to committed, updates timestamp/epoch. `discard()` transitions to retracted. Non-existent draft id rejects with `DRAFT_NOT_FOUND`. No in-place editing.

**RETRACT / SUPERSEDE:** non-author retract rejects with `RETRACT_NOT_AUTHORIZED`. Idempotent on already-retracted. Non-existent id rejects with `ENTRY_NOT_FOUND`. Intent meets minimum length. Any agent can supersede. Chain extension works. ATTUNE sees only the latest.

**Error codes:** all three new codes extend `AkashikError`. `instanceof AkashikError` works.

## Implementation notes

The closure-based factory pattern from v0.1 carries forward. State expansion: `entries` (existing), `drafts` (Map by draft_id), `sessions` (Map by agent id), `epochCounter` (number).

`findConflicts` runs over attune's filtered entries, not all entries. Relevance scoring computed fresh on every call — no caching in v0.2.

The wire envelope is constructed by an `envelope.ts` module that wraps every operation; the SDK's TypeScript surface stays clean.

Status transitions are direct property updates on the entry. The closure pattern guarantees external code can't observe mutation mid-transition.

## Open questions deferred to build phase

- Whether the wire envelope crosses a real transport in v0.2 or is constructed-but-unused. Current preference: constructed and validated, not crossing transport. Sets up Level 1+ transport bindings cleanly.
- Whether `relevance_reason.components` are individual contributions (sum to score) or normalised in some other way. Current preference: each component IS the contribution it added.
- Whether `discard()` requires intent. Current spec says yes (10-char minimum). Worth re-confirming during build.
- Whether SUPERSEDE's reference-to-predecessor is exposed in the entry shape or stays internal. Current preference: internal; surfaced via REPLAY in v0.3.

## Naming convention (carried forward)

Public methods are single, evocative verbs with deliberate gravity. The vocabulary register is contemplative and slightly mythological:

- v0.1: `write`, `read`, `attune`
- v0.2: `register`, `deregister`, `draft`, `commit`, `discard`, `retract`, `supersede`, `reckon`

`register`/`deregister` break the single-evocative-word pattern but match standard protocol vocabulary (HTTP, MCP, OAuth all use REGISTER). `commit`/`discard` are prosaic on purpose — lifecycle clarity over evocation. The contemplative register is preserved where it does work.

## Version policy

- `v0.1.x` — patch releases for bugs in v0.1's surface.
- `v0.2.0` — Level 0 conformance + selected Level 1 operations as defined here.
- `v0.2.x` — patch releases for v0.2's surface.
- `v0.3.0` — completes Level 1: persistence, event log, full Lamport, confidence, polling.
- Breaking changes still require a major version bump.
